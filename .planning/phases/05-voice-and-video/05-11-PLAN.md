Replace the `useRef + useEffect([stream])` pattern with a callback ref so `srcObject` is assigned the moment the `<video>` element mounts.

Root cause: when `cameraOn` flips to `true`, React adds the `<video>` element to the DOM for the first time. The `stream` reference is the same object it was (no change), so `useEffect([stream])` does not re-run. The video element therefore never gets `srcObject` assigned and renders gray.

Changes to make:

1. Remove `import { useEffect, useRef }` and replace with `import { useCallback, useRef }` — `useRef` is still needed for the screen share tile ref.

   Actually both tiles share one `videoRef` today, so the refactor should split them: the regular participant tile needs a callback ref; the screen share tile (which always has a stream when shown) can keep a plain `useRef`. Keep `useRef` in imports.

2. Add a `useCallback` callback ref for the regular tile video element:
   ```tsx
   const videoCallbackRef = useCallback(
     (el: HTMLVideoElement | null) => {
       if (el && stream) {
         el.srcObject = stream;
       }
     },
     [stream],
   );
   ```
   This fires whenever the element mounts OR whenever `stream` changes (because `useCallback` produces a new function when `stream` changes, and React calls the new ref function with the current element).

3. For the screen share tile, keep the existing `videoRef = useRef<HTMLVideoElement>(null)` and its `useEffect([stream])` — the screen share element is always present when the stream is non-null, so the timing issue does not apply there.

4. On the regular tile's `<video>` element, change `ref={videoRef}` to `ref={videoCallbackRef}`.

5. Remove the `useEffect` that was setting `videoRef.current.srcObject = stream` for the regular tile. The screen share tile's useEffect (if it had one) should remain; confirm it still calls `videoRef.current.srcObject = stream` in a `useEffect([stream])` block.

   Looking at the current code: there is one shared `useEffect([stream])` and one shared `videoRef`. After the split:
   - Regular tile: no useEffect for srcObject — handled by `videoCallbackRef`.
   - Screen share tile: add a dedicated `useEffect([stream])` that sets `videoRef.current.srcObject = stream` if both are non-null.

6. Update imports: remove `useEffect` if no longer needed elsewhere in the file; add `useCallback`.

Final import line should be:
```tsx
import { useCallback, useRef } from "react";
```

  <name>Task 2: Clamp PiP drag to all four viewport edges</name>
  <files>apps/client/src/components/voice/VoicePiP.tsx</files>
  <action>
Add upper-bound clamping in `handlePointerMove` so the PiP cannot be dragged off the top or right edges.

Root cause: the current code does `Math.max(0, ...)` which only prevents going off the bottom or left. There is no upper bound, so dragging up and to the right can move the PiP off-screen.

The PiP uses `position: fixed` with `right` and `bottom` CSS properties. The element width is 280px (hardcoded in inline style). The element height is not fixed — estimate ~120px, but we should read it from the container ref for accuracy.

Changes to `handlePointerMove`:

```tsx
const handlePointerMove = useCallback((e: React.PointerEvent) => {
  if (!draggingRef.current) return;

  const dx = e.clientX - dragStartRef.current.clientX;
  const dy = e.clientY - dragStartRef.current.clientY;

  if (Math.abs(dx) > 3 || Math.abs(dy) > 3) {
    hasDraggedRef.current = true;
  }

  const pipWidth = containerRef.current?.offsetWidth ?? 280;
  const pipHeight = containerRef.current?.offsetHeight ?? 120;

  const newRight = Math.min(
    window.innerWidth - pipWidth,
    Math.max(0, dragStartRef.current.right - dx),
  );
  const newBottom = Math.min(
    window.innerHeight - pipHeight,
    Math.max(0, dragStartRef.current.bottom - dy),
  );

  setPos({ right: newRight, bottom: newBottom });

  e.preventDefault();
}, []);
```

No other changes needed. The `containerRef` is already attached to the outer `<div>` in the JSX (`ref={containerRef}`).
