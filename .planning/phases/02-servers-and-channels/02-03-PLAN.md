  <action>
    1. Add invite types to `packages/shared/src/types/server.ts`:
       - `InviteResponse`: { id, serverId, creatorId, code, maxUses, uses, expiresAt, createdAt }
       - `CreateInviteRequest`: { expiresIn?: number (seconds), maxUses?: number | null }
       - `InviteInfoResponse`: { code, serverName, serverIcon, creatorName, memberCount, expiresAt } (for the invite preview page)

    2. Create `apps/server/src/routes/servers/invites.ts`:
       - GET `/:id/invites` — list all invites for a server
         - `preHandler: [fastify.authenticate]`
         - Verify the requesting user is a member of server `:id`
         - Return all invites for the server, joined with users table to include creator displayName
         - Return 200 with array

       - POST `/:id/invites` — create a new invite
         - `preHandler: [fastify.authenticate]`
         - Verify the requesting user is a member of server `:id`
         - Schema: body has optional `expiresIn` (integer, positive) and `maxUses` (integer, min 1, or null)
         - Generate invite code: `import { randomBytes } from "node:crypto"; randomBytes(8).toString("base64url")` (11-char URL-safe code)
         - Calculate expiresAt: if expiresIn provided, `new Date(Date.now() + expiresIn * 1000)`, else null (never expires)
         - Insert into `invites` table: { serverId, creatorId: request.user.id, code, maxUses: body.maxUses ?? null, expiresAt }
         - Return 201 with invite object

       - DELETE `/:id/invites/:inviteId` — revoke an invite
         - `preHandler: [fastify.authenticate]`
         - Verify the requesting user is the server owner (only owners can revoke invites)
         - Delete the invite row where id = inviteId AND serverId = id
         - If no row deleted, return 404
         - Return 200

    3. Create `apps/server/src/routes/invites/join.ts`:
       - GET `/:code` — invite preview (no auth required)
         - Look up invite by code, join with servers table to get server name, icon, member count
         - Check expiry and max uses — if invalid, return 410 Gone with { error: "Invite is expired or has reached its use limit" }
         - Return 200 with InviteInfoResponse (does NOT consume a use)

       - POST `/:code/join` — join a server via invite code
         - `preHandler: [fastify.authenticate]`
         - ATOMIC increment: Use a single UPDATE with WHERE conditions:
           ```typescript
           const [invite] = await db
             .update(invites)
             .set({ uses: sql`${invites.uses} + 1` })
             .where(
               sql`${invites.code} = ${code}
                 AND (${invites.maxUses} IS NULL OR ${invites.uses} < ${invites.maxUses})
                 AND (${invites.expiresAt} IS NULL OR ${invites.expiresAt} > NOW())`
             )
             .returning();
           ```
         - If no row returned: invite is expired/exhausted -> 410 Gone
         - Check if user is already a member of invite.serverId -> if yes, return 409 Conflict { error: "Already a member" } (and decrement uses back, or do the membership check BEFORE the atomic update)
         - BETTER approach: check membership FIRST, then do the atomic update:
           a. Check `serverMembers` for existing membership -> 409 if exists
           b. Atomic UPDATE invites with uses+1 and WHERE conditions -> 410 if no row
           c. Insert into `serverMembers` { serverId: invite.serverId, userId: request.user.id }
         - Broadcast `fastify.io.to(\`server:${invite.serverId}\`).emit("member:joined", { serverId: invite.serverId, userId: request.user.id })`
         - Return 200 with { server: { id, name, ... } } so the client can navigate to the server

    4. Register in `apps/server/src/index.ts`:
       ```
       import serverInvitesRoute from "./routes/servers/invites.js";
       import inviteJoinRoute from "./routes/invites/join.js";

       await server.register(serverInvitesRoute, { prefix: "/api/servers" });
       await server.register(inviteJoinRoute, { prefix: "/api/invites" });
       ```

    IMPORTANT: The atomic UPDATE WHERE pattern from RESEARCH.md Pitfall 3 is critical. Do NOT read-then-write. The membership check before the atomic update is acceptable because the worst case is a user who is already a member consuming a use — but since we check first, this is prevented. Wrap the check + update + insert in a transaction for consistency.
