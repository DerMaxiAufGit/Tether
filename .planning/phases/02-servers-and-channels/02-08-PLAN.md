  <action>
    Three surgical fixes:

    **1. useServers.ts — Add `exact: true` to invalidateQueries (Test 2):**

    In the `useCreateServer` hook (line 21), change:
    ```ts
    void queryClient.invalidateQueries({ queryKey: ["servers"] });
    ```
    to:
    ```ts
    void queryClient.invalidateQueries({ queryKey: ["servers"], exact: true });
    ```

    WHY: Without `exact: true`, TanStack Query uses prefix matching. `["servers"]` matches `["servers"]` AND `["servers", serverId, "channels"]`. When CreateServerModal navigates to the new server immediately after creation, ChannelList mounts and starts fetching channels, but the prefix invalidation also invalidates that query at the same moment, causing `isLoading` to stay true (infinite loading). `exact: true` only invalidates the server list query, not channel queries.

    **2. CreateServerModal.tsx — Emit server:subscribe after creation (Test 10):**

    Import `useSocket` at the top:
    ```ts
    import { useSocket } from "@/hooks/useSocket";
    ```

    Get the socket instance inside the component:
    ```ts
    const socket = useSocket();
    ```

    In the `handleCreate` function, after `createServer.mutateAsync` succeeds and before `handleClose()`, emit `server:subscribe` so the socket joins the new server's room for real-time events:
    ```ts
    const server = await createServer.mutateAsync({ name });
    const serverId = (server as ServerResponse).id;

    // Join the socket room for the new server so real-time events work
    socket.emit("server:subscribe", { serverId });

    handleClose();
    navigate(`/servers/${serverId}`);
    ```

    WHY: After creating a server, the user's socket is not in the new server's room (it was created after the initial subscription). Without subscribing, events like `server:created` from other tabs or `channel:created` within the server are never received. The server-side `server:subscribe` handler verifies DB membership before joining the room, so this is safe.

    **3. useSocket.tsx — Add reconnect_attempt handler to refresh token (Test 10):**

    In the connection lifecycle `useEffect` (around line 71), after `socket.connect()`, add a listener for `reconnect_attempt` that refreshes the auth token before each reconnect attempt:

    ```ts
    // Refresh auth token before each reconnect attempt
    const onReconnectAttempt = () => {
      const freshToken = getAccessToken();
      socket.auth = { token: freshToken };
    };
    socket.io.on("reconnect_attempt", onReconnectAttempt);
    ```

    And in the cleanup function, remove the listener:
    ```ts
    return () => {
      socket.io.off("reconnect_attempt", onReconnectAttempt);
      socket.disconnect();
    };
    ```

    Note: `reconnect_attempt` is on `socket.io` (the Manager), not on `socket` itself.

    WHY: The socket's auth token is set once at connect time. If the socket disconnects and reconnects (e.g., network blip, server restart), the original token may have expired. Without refreshing, the reconnect silently fails authentication, causing the socket to disconnect permanently. The user never receives real-time events again until a full page refresh.
