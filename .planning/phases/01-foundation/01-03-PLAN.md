    1. Create packages/shared/src/types/crypto-worker.ts with message types for worker communication:

       Define the request message types (main thread -> worker):
       - DERIVE_KEYS: { password: string, salt: string } (salt is base64) -> returns { authKey: string, encryptionKey stays internal }
       - GENERATE_AND_WRAP_KEYPAIR: { encryptionKey: <internal reference> } -> returns full key bundle
       - REGISTER: { password: string } -> runs full registration flow (derive + generate + wrap), returns everything the server needs
       - LOGIN_DECRYPT: { password: string, salt: string, x25519Blob: string, x25519Iv: string, ed25519Blob: string, ed25519Iv: string } -> decrypts private keys, returns authKey + signals success
       - CHANGE_PASSWORD: { currentPassword: string, newPassword: string, currentSalt: string, x25519Blob: string, x25519Iv: string, ed25519Blob: string, ed25519Iv: string } -> re-derives, decrypts with old, re-encrypts with new, returns new blobs + new authKey + new salt

       Define the response message types (worker -> main thread):
       - PROGRESS: { step: "deriving" | "generating" | "encrypting" | "decrypting" | "re-encrypting" | "done" }
       - RESULT: { success: true, data: <operation-specific result> }
       - ERROR: { success: false, error: string }

       All binary data is transmitted as base64 strings (postMessage can handle ArrayBuffer via transferable, but base64 is simpler and the data is small — key blobs are < 1KB).

       Use discriminated unions with a `type` field for type-safe message matching.

    2. Update packages/shared/src/index.ts to re-export crypto-worker types.

    3. Create apps/client/src/workers/crypto.worker.ts implementing all operations using ONLY the Web Crypto API (NO npm crypto libraries):

       **Helper functions (internal to worker):**

       a) `deriveKeysFromPassword(password: string, salt: Uint8Array)`:
          - Import password as PBKDF2 key material
          - PBKDF2 deriveBits: 600,000 iterations, SHA-256, 512 bits output
          - Import result as HKDF key material
          - HKDF deriveBits with info "tether-auth-key-v1" -> 256-bit auth key
          - HKDF deriveKey with info "tether-encryption-key-v1" -> AES-GCM CryptoKey (NOT extractable)
          - Import constants (KDF_ITERATIONS, AUTH_HKDF_INFO, ENCRYPTION_HKDF_INFO) from @tether/shared
          - Return { authKey: Uint8Array, encryptionKey: CryptoKey }

       b) `generateAndWrapKeypair(encryptionKey: CryptoKey)`:
          - Generate X25519 keypair (extractable: true, ["deriveKey", "deriveBits"])
          - Generate Ed25519 keypair (extractable: true, ["sign", "verify"])
          - Export public keys: X25519 as "raw", Ed25519 as "raw" (use "raw" for both — Ed25519 raw export is the 32-byte public key; if "raw" doesn't work for Ed25519, use "spki" and document)
          - Export private keys as "pkcs8"
          - Encrypt each private key with AES-GCM using the encryptionKey, random 12-byte IV
          - Return { x25519Public, ed25519Public, x25519EncryptedPrivate: {ciphertext, iv}, ed25519EncryptedPrivate: {ciphertext, iv} }

       c) `unwrapPrivateKeys(encryptionKey: CryptoKey, x25519Blob, x25519Iv, ed25519Blob, ed25519Iv)`:
          - Decrypt each private key blob with AES-GCM
          - Import X25519 private key from PKCS8 (["deriveKey", "deriveBits"])
          - Import Ed25519 private key from PKCS8 (["sign"])
          - Return the CryptoKey objects (these stay in the worker for future use)
          - NOTE: For Phase 1, we just prove decryption works. The keys will be stored in worker memory for Phase 3 use.

       d) `uint8ToBase64(bytes: Uint8Array): string` and `base64ToUint8(base64: string): Uint8Array` — conversion helpers.

       **Main message handler (self.onmessage):**

       Handle REGISTER:
       1. Post progress: "deriving"
       2. Generate 32-byte random salt
       3. Call deriveKeysFromPassword
       4. Post progress: "generating"
       5. Call generateAndWrapKeypair
       6. Post progress: "encrypting" (already done in step 5, but the step label makes UX sense)
       7. Post progress: "done"
       8. Return: { authKey (base64), salt (base64), x25519PublicKey (base64), ed25519PublicKey (base64), x25519EncryptedPrivateKey (base64), x25519KeyIv (base64), ed25519EncryptedPrivateKey (base64), ed25519KeyIv (base64) }

       Handle LOGIN_DECRYPT:
       1. Post progress: "deriving"
       2. Call deriveKeysFromPassword with provided salt
       3. Post progress: "decrypting"
       4. Call unwrapPrivateKeys — if decryption fails (wrong password), the AES-GCM decrypt will throw (GCM tag mismatch)
       5. Post progress: "done"
       6. Return: { authKey (base64), success: true }
       - If AES-GCM throws, return error "Decryption failed — incorrect password"

       Handle CHANGE_PASSWORD:
       1. Post progress: "deriving"
       2. Derive keys from current password + current salt
       3. Post progress: "decrypting"
       4. Decrypt existing private key blobs with current encryption key
       5. Post progress: "re-encrypting"
       6. Generate new 32-byte salt, derive keys from new password + new salt
       7. Re-encrypt private keys with new encryption key (new random IVs)
       8. Post progress: "done"
       9. Return: { newAuthKey (base64), oldAuthKey (base64), newSalt (base64), x25519EncryptedPrivateKey (base64), x25519KeyIv (base64), ed25519EncryptedPrivateKey (base64), ed25519KeyIv (base64) }
       - oldAuthKey is needed so server can verify current password before accepting the change.

       IMPORTANT: Use `import.meta.url` compatible Vite worker pattern. The worker file must be a module worker (type: 'module') so it can import from @tether/shared.

    2. Implements a request/response pattern with correlation IDs:
       - Each call generates a unique ID (crypto.randomUUID())
       - Posts message to worker with the ID
       - Returns a Promise that resolves when the worker responds with a matching ID

    3. Provides a progress callback mechanism:
       - Each method accepts an optional `onProgress?: (step: string) => void` callback
       - PROGRESS messages from the worker are forwarded to this callback
       - The Promise only resolves on the final RESULT message

    4. Exports these typed functions:

       ```typescript
       export interface RegisterResult {
         authKey: string;      // base64
         salt: string;         // base64
         x25519PublicKey: string;
         ed25519PublicKey: string;
         x25519EncryptedPrivateKey: string;
         x25519KeyIv: string;
         ed25519EncryptedPrivateKey: string;
         ed25519KeyIv: string;
       }

       export interface LoginDecryptResult {
         authKey: string;      // base64
       }

       export interface ChangePasswordResult {
         oldAuthKey: string;   // base64 — for server verification
         newAuthKey: string;   // base64
         newSalt: string;      // base64
         x25519EncryptedPrivateKey: string;
         x25519KeyIv: string;
         ed25519EncryptedPrivateKey: string;
         ed25519KeyIv: string;

       export function register(
         password: string,
         onProgress?: (step: string) => void,

         onProgress?: (step: string) => void,
       ): Promise<ChangePasswordResult>;
       ```
