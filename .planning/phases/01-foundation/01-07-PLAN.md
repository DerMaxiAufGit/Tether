    3. Create apps/client/src/pages/auth/LoginPage.tsx:
       - Wrapped in AuthLayout
       - Form fields: email, password
       - Link to /register for new users
       - On submit:
         a. First, call POST /api/auth/login with just the email to get the salt and key bundle
            - WAIT: Actually the login flow needs the auth key. So the client needs the salt FIRST to derive keys. But the salt is stored server-side.
            - CORRECTED FLOW:
              1. Client sends POST /api/auth/login/salt with { email } to get { salt } (public, no auth needed)
              2. Client derives auth key from password + salt using crypto worker
              3. Client sends POST /api/auth/login with { email, authKey }
              4. Server verifies, returns access token + key bundle
              5. Client decrypts private keys using password + salt (already derived encryption key)
            - OR: Simpler approach — server returns salt + key bundle on login attempt, client derives locally. This means the login endpoint:
              1. POST /api/auth/login with { email, password: <not sent>, authKey: <not sent yet> }
              -- This doesn't work either because we need the salt first.
            - FINAL APPROACH (two-step login):
              Step 1: POST /api/auth/challenge with { email } -> returns { salt } (or 401 if no user — but this leaks email existence; return fake salt for non-existent users to avoid timing attack)
              Step 2: Client derives authKey + encryptionKey from password + salt in Web Worker
              Step 3: POST /api/auth/login with { email, authKey } -> returns { accessToken, user, keyBundle }
              Step 4: Client decrypts private keys from keyBundle using encryptionKey (already in Worker memory)
         b. Show KeyDerivationProgress during step 2
         c. On success: store auth state, navigate to /
         d. On error: show "Invalid email or password" (generic, no email existence leak beyond the challenge step which uses fake salts)
